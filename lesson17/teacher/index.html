エラーの種類とtry/catchを理解しよう
0〜25分
💬
講師がこう話す「プログラムを書いているとエラーは必ず起きます。大切なのはエラーが起きたときにプログラムが壊れないようにすること、そしてエラーの原因を素早く特定することです。try/catchはエラーをキャッチして適切に対処するための構文です。」
🐛
JavaScriptの主なエラーの種類
SyntaxError：構文エラー。スペルミス・括弧の対応ミスなど。コードが実行される前に発生。
ReferenceError：定義されていない変数を参照したとき。
TypeError：値の型が期待と異なるとき（nullのプロパティを参照するなど）。
RangeError：値が有効な範囲外のとき（配列の長さに負の値など）。
NetworkError：fetch失敗など通信エラー。
script.js（try/catch/finally）
// try：エラーが起きるかもしれない処理を囲む // catch：エラーが起きたときに実行される // finally：成功・失敗に関わらず必ず実行される try { const data = JSON.parse('{"name": "太郎"}'); // 正常 console.log(data.name); const broken = JSON.parse('不正なJSON'); // ← エラー発生 console.log("これは表示されない"); } catch (error) { // errorオブジェクトには情報が詰まっている console.log(error.name); // "SyntaxError" console.log(error.message); // エラーの詳細メッセージ } finally { console.log("finallyは必ず実行される"); // ローディング解除・リソースの後片付けに使う }
script.js（throwで意図的にエラーを発生させる）
// throwで自分でエラーを投げられる const divide = (a, b) => { if (b === 0) { throw new Error("0で割ることはできません"); } return a / b; }; try { console.log(divide(10, 2)); // 5 console.log(divide(10, 0)); // ← throwされる } catch (error) { console.log(error.message); // "0で割ることはできません" } // 文字列・数値・オブジェクトも throw できる throw { code: 404, message: "Not Found" };
try/catchを使うべき場面
全部囲めばいいわけではない
✅ 使うべき場面：外部からのデータ（API・JSON・ユーザー入力）を扱うとき、ファイル操作やDB処理、外部ライブラリの呼び出し。

❌ 使わなくていい場面：自分が完全にコントロールしている計算処理、エラーが起きえないシンプルな関数。

「エラーが起きる可能性がある場所だけ囲む」が原則です。
👆
講師メモtry/catchを何でもかんでも囲む生徒がいる。「エラーを隠す道具ではなく、エラーを適切に処理する道具」だと伝える。
❓

カスタムエラー・デバッグ術
25〜45分
💬
講師がこう話す「Errorクラスを継承してカスタムエラーを作ると、エラーの種類を判別しやすくなります。また、console.error・console.warn・console.table・デベロッパーツールのブレークポイントなどデバッグの武器を紹介します。」
script.js（カスタムエラークラス）
// Errorを継承してカスタムエラーを作る class ValidationError extends Error { constructor(message, field) { super(message); this.name = "ValidationError"; this.field = field; } } class NetworkError extends Error { constructor(message, status) { super(message); this.name = "NetworkError"; this.status = status; } } // instanceof でエラーの種類を判別して処理を分岐 try { throw new ValidationError("メールアドレスが不正です", "email"); } catch (error) { if (error instanceof ValidationError) { console.log(`バリデーションエラー [${{error.field}}]: ${{error.message}}`); } else if (error instanceof NetworkError) { console.log(`通信エラー (${{error.status}}): ${{error.message}}`); } else { throw error; // 想定外のエラーは再度throwする } }
script.js（デバッグに使えるconsoleメソッド）
// console.log → 通常の出力 // console.error → 赤いエラー表示 // console.warn → 黄色い警告表示 // console.table → 配列・オブジェクトを表形式で表示 // console.group → グループ化して折りたためる // console.time → 処理時間を計測 const users = [ { name: "太郎", age: 25 }, { name: "花子", age: 22 } ]; console.table(users); // 表形式で見やすく表示 // 処理時間を計測 console.time("処理時間"); for (let i = 0; i < 100000; i++) {} // 何か処理 console.timeEnd("処理時間"); // "処理時間: Xms" // グループ化 console.group("ユーザー情報"); console.log("名前: 太郎"); console.log("年齢: 25"); console.groupEnd();
💡
デベロッパーツールのSourcesタブでブレークポイントを設定すると、コードを1行ずつ実行して変数の中身を確認できます。console.logだらけのデバッグより遥かに効率的です。ぜひ習慣にしましょう。
